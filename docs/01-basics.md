# 01. Basics And Control Loop

## Goals

- understand where Nginx sits in the request path
- learn the main config contexts: `http`, `server`, and `location`
- practice the config test and reload workflow

## Files To Look At

- `nginx/nginx.conf`
- `nginx/conf.d/20-base-server.conf`
- `nginx/conf.d/30-host-routing.conf`

## Mental Model

Think about a single request coming into Nginx on port `80`.

1. The request arrives on a listening socket.
2. Nginx chooses a `server` block based on the `Host` header and listen port.
3. Inside that server, Nginx chooses the best `location` block.
4. Nginx either serves a file or proxies the request to an upstream.
5. Logs record what happened, including request time and upstream timing.

## Start The Stack

```bash
docker compose up -d --build
```

## Inspect The Live Config Workflow

Test the config:

```bash
docker compose exec nginx nginx -t
```

Reload the config:

```bash
docker compose exec nginx nginx -s reload
```

Watch logs:

```bash
docker compose logs -f nginx
```

## Verify

```bash
curl -sS http://localhost:8080/
curl -sS http://localhost:8080/healthz
```

What to observe:

- `/` is served directly by Nginx from the mounted static files.
- `/healthz` is generated by the Nginx config itself, not by an upstream app.
- log lines include upstream fields even when a request did not hit an upstream.
